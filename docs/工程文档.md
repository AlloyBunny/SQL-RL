# SQL-RL 工程文档

## 1. 项目结构

计划目录结构：

```text
SQL-RL/
  docs/
    设计文档.md
    工程文档.md
    环境配置文档.md
  sql_rl/
    data/
    prompts/
    env/
    rewards/
    eval/
    train/
    utils/
  configs/
  scripts/
  outputs/
  logs/
```

暂时只创建 `docs/`。代码目录在开始实现时再创建。

## 2. 模块划分

`sql_rl/data/`

负责读取 BIRD 或 BIRD-like 数据，生成统一训练样本格式。

`sql_rl/prompts/`

负责 prompt 模板和 chat messages 构造。

`sql_rl/env/`

负责 SQLite schema 提取、SQL 执行、超时控制、结果标准化。

`sql_rl/rewards/`

负责 correctness reward 和 cost-aware reward。

`sql_rl/eval/`

负责 baseline、SFT、RL 模型的执行准确率评估。

`sql_rl/train/`

负责训练框架适配。第一版优先适配 ms-swift，必要时增加 TRL adapter。

## 3. 数据格式

内部统一样本格式：

```json
{
  "id": "sample_id",
  "db_id": "database_id",
  "question": "natural language question",
  "schema": "database schema text",
  "evidence": "optional evidence",
  "gold_sql": "reference SQL",
  "db_path": "path/to/database.sqlite",
  "difficulty": "optional difficulty"
}
```

训练框架需要什么格式，由 adapter 转换，不污染核心数据结构。

## 4. Prompt 格式

第一版使用简单稳定格式：

```text
You are an expert SQL generator.
Given a database schema and a question, write a valid SQLite SQL query.
Return only the SQL query.
```

后续可以增加 reasoning 格式，但第一版优先保证 reward 解析简单。

如果使用 `<solution>...</solution>` 格式，需要在 reward 里稳定解析 SQL。

## 5. SQLite 执行器

执行器需要支持：

- 执行预测 SQL。
- 执行 gold SQL。
- 设置超时时间。
- 捕获 SQL 语法错误和执行错误。
- 标准化结果用于比较。
- 记录 error type，便于后续分析。

第一版只处理 SELECT 查询，不处理会修改数据库状态的 SQL。

## 6. Reward 模块

基础 reward：

```text
score = 1.0 if normalized(pred_result) == normalized(gold_result) else 0.0
```

cost-aware reward：

```text
score = correctness
        - lambda_len * length_penalty
        - lambda_invalid * invalid_penalty
        - lambda_timeout * timeout_penalty
```

reward 函数应框架无关。训练框架 adapter 只负责把模型输出和 batch 信息传进来。

## 7. Evaluation 模块

Evaluation 需要输出：

- 总样本数。
- EX accuracy。
- invalid SQL rate。
- timeout rate。
- average SQL length。
- 错误样例 JSONL。

评估结果保存到：

```text
outputs/eval/
```

## 8. 训练后端

第一选择：ms-swift。

理由：

- 使用者更熟悉。
- 对 Qwen 系模型训练更方便。
- 支持 SFT 和 GRPO 类训练流程。
- 自定义 reward 接入成本预计低于 veRL。

备选：TRL。

理由：

- Python 代码透明。
- GRPOTrainer 和 reward function 逻辑容易调试。

暂不使用：veRL。

理由：

- 环境复杂。
- 对单轮 SQL RLVR 不是必要条件。
- BIRD-RL 复杂部分主要集中在 veRL、vLLM、Ray、agent loop 和 patch。

## 9. 开发顺序

第一阶段：文档和项目结构。

第二阶段：数据读取、prompt 构造、SQLite 执行器。

第三阶段：baseline evaluation。

第四阶段：SFT。

第五阶段：GRPO / RLVR。

第六阶段：cost-aware reward ablation。

第七阶段：整理实验结果和简历表述。

## 10. 最小可行闭环

第一版必须先跑通：

```text
读取样本 -> 构造 prompt -> 模型生成 SQL -> SQLite 执行 -> 计算 reward -> 保存评估结果
```

训练可以晚于 evaluation。只要 evaluation 和 reward 模块先稳定，后续换训练框架的风险会小很多。
