# SQL-RL 设计文档

## 1. 项目定位

本项目是一个面向 post-training 实习展示的 SQL 强化学习项目。目标不是完整复现 BIRD-RL，而是基于 BIRD-RL 的数据定位和执行验证思路，构建一个轻量、可解释、可复现实验的 `SQL-RL`。

核心方向调整为 **SQL Debugging / SQL Repair + RLVR**：

```text
User issue + buggy SQL + database schema
-> LLM repairs SQL
-> SQLite execution + test-case validation
-> reward
-> GRPO update
```

这个定位比单纯 Text-to-SQL 更贴合 BIRD-RL 的主线数据：`birdsql/six-gym-sqlite` 的样本核心字段是 `query + issue_sql -> sol_sql`，并带有 `test_cases` 和 SQLite 数据库文件，天然适合 execution-verifiable RL。

## 2. 总体目标

完整项目希望覆盖下面几层能力：

1. 基础 SQL debugging / repair 能力。
2. 基于 SQLite 执行结果和 test case 的可验证 reward。
3. SFT -> GRPO / RLVR 的后训练闭环。
4. 面向效率和稳定性的 reward shaping。
5. 数据划分、评测、失败样例分析。
6. 可选的轻量多轮修复能力。

其中前 1-5 项是主线，轻量多轮修复是增强能力，不应该阻塞第一版闭环。

## 3. 主线任务

主线任务是 `SQL Debugging + RLVR`。

任务输入：

```text
query + issue_sql + database schema + optional preprocess context
```

任务输出：

```text
fixed SQL
```

验证方式：

```text
在对应 SQLite 数据库上执行模型 SQL，再运行 test_case。
test_case 通过则 reward=1，否则 reward=0。
```

这条主线的价值在于：reward 不依赖主观 judge，而是直接由数据库执行和 Python test case 判定。它更像 SQL 版力扣：模型提交 SQL，系统在真实数据库状态上运行测试。

## 4. 数据选择

第一版使用：

- 训练/自建验证：`birdsql/six-gym-sqlite`。
- 外部评测/展示：`birdsql/bird-critic-1.0-sqlite`，但 public 版本不带 `sol_sql/test_cases`，只能做执行成功率、案例分析，或在拿到 full data 后做正式自动评测。

不优先使用 BIRD 官网 train/dev。它们是原始 BIRD-SQL Text-to-SQL 数据，任务形态是 `question + schema -> SQL`，和本项目主线的 SQL debugging/repair 不一致，且 train 包约 8.92GB，对第一版个人项目没有必要。

详细数据说明见 [数据文档.md](./数据文档.md)。

## 5. 训练闭环

完整训练闭环设计为：

```text
SFT cold start -> GRPO / RLVR -> reward ablation -> evaluation
```

其中：

- SFT 用 `sol_sql` 作为监督信号，让模型学会从 `query + issue_sql` 修复 SQL。
- RL 阶段使用 test-case based reward。
- reward ablation 用来验证 cost-aware shaping 是否有效。
- evaluation 量化最终能力，而不是只看训练日志。

GRPO 的关键点包括：

- group sampling，对同一个 SQL debugging 样本采样多个修复结果。
- relative advantage，使用组内相对优势更新。
- 可验证 reward，直接基于 SQLite 执行和 test case。

## 6. SFT 设计

SFT 阶段的目标不是追求复杂 agent，而是让模型先学会输出可执行、语义正确的修复 SQL。

数据来源：

- `birdsql/six-gym-sqlite/train.jsonl`。

训练形式：

```text
query + issue_sql + schema -> sol_sql
```

推荐输出格式：

```text
<solution>
fixed SQL
</solution>
```

使用 `<solution>` 的原因是 reward 阶段需要稳定抽取 SQL。第一版可以不强制模型输出长 reasoning，避免解析复杂度影响 reward 稳定性。

## 7. RL 设计

RL 阶段使用 SQL execution 和 test case 作为可验证 reward。

基础 reward：

```text
R_correct = 1 if all test_cases pass else 0
```

更细的版本：

```text
R_correct = test_pass_count / test_total_count
```

RL 阶段需要抽象出：

- prompt 构造。
- SQL 解析。
- SQLite 数据库准备和隔离。
- `preprocess_sql` 执行。
- 模型 SQL 执行。
- `test_cases` 执行。
- reward 计算。
- evaluation 日志。

训练后端可以选择 ms-swift 或 TRL，但 reward/eval 核心逻辑应尽量框架无关。

## 8. 核心改进点

项目不能只停留在“复现 GRPO”，必须有自己的改动点。第一版推荐 **cost-aware reward**。

基础 reward：

```text
R_correct = 1 if test_cases pass else 0
```

cost-aware reward：

```text
R = R_correct
    - lambda_len * normalized_sql_length
    - lambda_invalid * invalid_sql
    - lambda_timeout * timeout
```

如果做轻量多轮修复，还可以加入：

```text
R = R_correct
    - lambda_call * execution_count
    - lambda_retry * repair_turns
```

目标是在尽量保持修复成功率的同时，降低无效 SQL、超时、过长 SQL 和不必要执行次数。

## 9. 数据效率和任务分层

除了 reward shaping，还可以加入数据效率实验：

- full dataset。
- 10% / 30% subset。
- 按 `category` 分层：Query、Management、Personalization。
- 按是否含 `preprocess_sql` 分层。

这个方向适合证明模型不是单纯靠数据堆出来的，而是通过任务结构、reward 设计和采样策略提高训练效率。

## 10. 可选增强

### 10.1 轻量多轮修复

第一版主线仍然建议单轮：

```text
query + issue_sql -> fixed SQL
```

后续可以允许模型在执行失败后重新修复 1-2 次：

```text
fixed SQL -> execution error/test failure -> revised SQL
```

这比完整 agent tool loop 更轻，适合个人项目作为增强点。

### 10.2 Text-to-SQL 对照实验

Text-to-SQL 不作为主线，但可以作为对照或附录实验。如果后续时间充足，可以用 BIRD-SQL mini/dev 做一个小规模 generation baseline，用来说明本项目选择 SQL debugging 的原因。

## 11. 实验设计

完整实验至少应包含以下对比：

- Base model zero-shot / few-shot SQL repair。
- SFT model。
- SFT + GRPO with correctness-only reward。
- SFT + GRPO with cost-aware reward。
- 可选：数据效率采样对比。
- 可选：轻量多轮修复对比。

主要指标：

- Success Rate / test-case pass rate。
- Execution success rate。
- Invalid SQL rate。
- Timeout rate。
- 平均 SQL 长度。
- 平均执行次数。
- reward 曲线。
- 训练稳定性。
- 样本效率。

## 12. 与 BIRD-RL 的关系

BIRD-RL 是参考来源，不是本项目的直接运行依赖。

需要借鉴的部分：

- SQL debugging 数据定义。
- prompt 构造思路。
- SQLite schema 提取方式。
- test-case based reward。
- persistent / isolated database execution 的工程思路。
- evaluation 思路。

不作为主依赖的部分：

- `bird_rl.yaml`。
- veRL patch。
- Ray / vLLM / 多机 rollout。
- 完整四阶段脚本。
- 完整多轮 agent tool loop。

也就是说，本项目保留 BIRD-RL 的方法论和任务定位，但不继承它的全部工程负担。

## 13. 退路

如果完整 RL 跑不通，项目仍然成立，且有层级可退：

1. 只做 dataset processing + baseline evaluation。
2. SFT + evaluation。
3. SFT + single-turn GRPO。
4. 再逐步加入 cost-aware reward。

这意味着项目不会因为某一层工程失败而整体失效。

## 14. 简历表达

项目最终希望能表达成：

```text
Built a SQL debugging RLVR pipeline that trains LLMs to repair faulty SQL using execution-verified rewards.
Implemented SFT + GRPO on BIRD-Critic/SIX-GYM SQLite tasks, with test-case based reward and cost-aware reward shaping to improve repair success while reducing invalid SQL, timeout, and inference cost.
```

重点不是“用了什么训练框架”，而是能讲清楚 SQL debugging 任务定义、数据库执行隔离、test-case reward、GRPO 训练闭环、reward shaping 和失败样例分析。
